# Project Execution Rules

이 문서는 `/Users/jungsunghoon/cctv-monitor` 작업 시 기본 실행 원칙을 고정하기 위한 문서다.

## Core Rule

모든 작업은 아래 순서로 진행한다.

1. 요구사항을 짧게 고정한다.
2. 제약과 금지 조건을 먼저 확인한다.
3. 가장 작은 안전 단위로 바로 실행한다.
4. 결과를 검증한다.
5. 다음 한 단계만 남긴다.

## Start Harness

작업 시작 전 내부적으로 아래 3가지를 먼저 확정한다.

- 문제 정의: 지금 실제로 풀어야 하는 문제 1개
- 완료 조건: 끝났다고 말할 수 있는 상태 1개 이상
- 금지 조건: 하면 안 되는 것, 거짓 정합, 임의 추정, 과한 자동 승격

이 단계에서 불필요한 배경 설명은 줄이고 바로 실행으로 들어간다.

## Token Discipline

토큰 낭비를 줄이기 위해 아래를 지킨다.

- 이미 확인한 사실은 반복 설명하지 않는다.
- 추측보다 기존 파일, 기존 로그, 기존 API 결과를 우선 사용한다.
- 넓게 탐색하지 말고 현재 작업에 필요한 범위만 읽는다.
- 같은 검증을 반복하지 않는다. 필요하면 스크립트화한다.
- 중간 보고는 `핵심 변화 / 검증 결과 / 다음 한 단계`만 말한다.
- 길게 설명하기보다 결과와 근거 파일을 먼저 제시한다.

## Execution Style

- 가능한 한 바로 구현한다.
- 막히면 먼저 로컬 문맥과 기존 산출물을 소진한다.
- 질문은 꼭 필요한 경우에만 짧게 한다.
- 자동화 가능한 작업은 스크립트, 리포트, 체크리스트로 남긴다.
- 수동 검토가 필요한 경우 자동 승격보다 `review_needed`를 우선한다.

## Safety Rule

이 프로젝트에서는 아래 원칙을 유지한다.

- 거짓 정합보다 숨김이 낫다.
- 근거 없는 좌표 승격은 금지한다.
- 근거 없는 중복 제거는 금지한다.
- 운영 노출 데이터는 `official` 또는 `verified` 중심으로 유지한다.
- 애매한 데이터는 `pending`, `review_needed`, `keep_hidden`으로 남긴다.

## Reporting Format

중간 보고는 아래 형식으로 최대한 짧게 유지한다.

- 무엇을 바꿨는지
- 무엇으로 검증했는지
- 지금 남은 다음 한 단계가 무엇인지

최종 보고는 아래 순서를 우선한다.

1. 결과
2. 검증
3. 남은 리스크 또는 다음 단계

## Verification Rule

가능하면 아래 순서로 검증한다.

1. 타입/빌드 검증
2. 로컬 스크립트 출력 검증
3. 운영 API 확인
4. 운영 UI 확인

한 번 만든 검증은 다음에도 재사용할 수 있게 남긴다.

## Review Workflow Rule

좌표, 매핑, 반자동 승격 작업은 아래 상태를 유지한다.

- `active`: 운영 반영 가능
- `review_needed`: 검토 대기
- `pending`: 근거 부족
- `keep_hidden`: 운영 숨김 유지

자동 승격은 높은 precision이 입증된 경우에만 쓴다.

## Browser Rule

브라우저를 띄울 일이 있으면 기본은 `Safari`를 사용한다.

## Preference Rule

이 프로젝트에서는 아래 성향을 기본값으로 둔다.

- 짧고 단단하게 진행
- 요구사항과 완료 조건을 먼저 고정
- 기능을 적극적으로 쓰되 과한 탐색은 피함
- 설명보다 실행과 검증 우선

## Update Rule

이 문서는 작업 방식이 바뀌면 바로 갱신한다.
